# Pokemon Champions Assistant — PRD

## 1. Executive Summary

Pokemon Champions Assistant 是一个面向**主播 / 内容创作者**的宝可梦对战**全流程辅助工具**。

它运行在电脑端，配合 OBS 虚拟摄像头从 Switch 对战画面中获取信息，在**独立浏览器窗口**中实时追踪每一场对局的全生命周期——从选人开始，到结算结束——自动识别双方队伍、场上宝可梦、HP、技能使用等变化，并持续生成对战记录。

长期目标包括但不限于：
- 阵容推荐器
- 伤害计算器
- 对战决策助手
- 宝可梦资料检索
- 战术建议系统
- 多语言视频信号识别能力

MVP 已覆盖识别链路、全流程追踪、5 列 UI 布局、截图/识别分离等核心能力。

---

## 2. Mission

打造一个适合直播场景的宝可梦对战辅助系统，让主播在观看对战画面时，能够通过旁路工具窗口**完整追踪每一场对局**——从选人到结算不遗漏任何关键信息。

核心使命：
- **缩短主播检索信息的时间**
- **降低直播中临场决策压力**
- **为后续自动识别和实时决策打下基础**

## 3. Target Users

### Primary Users
- 宝可梦对战主播
- 游戏内容创作者

### User Characteristics
- 通常会在电脑上同时打开 OBS、浏览器、聊天工具等多个窗口

- 希望辅助工具尽量少打断直播节奏
- 对信息呈现的**清晰度、速度、稳定性**要求高于花哨 UI

## 4. 全流程生命周期

Pokemon Champions Assistant 的核心设计原则是**追踪对局的完整生命周期**：

```
[画面信号正常] ──→ [选人开始] ──→ [战斗开始] ──→ [结算画面] ──→ [清除数据]
                           ↓                  ↓                  ↓
                    识别双方 6 只队伍     持续追踪 HP/技能/更换   准备下一局
```

### 生命周期规则
1. **开始条件**：检测到选人界面（`team_select` 阶段），视为对局开始
2. **进行中**：任何有对战内容的阶段（选人、战斗）均视为对局未结束
3. **结束条件**：检测到结算画面（获胜/失败）才是明确的结束
4. **未结束判断**：阶段不明（`unknown`）的截图**不**代表对局结束，需继续追踪；只有进入结算才清除
5. **数据持久原则**：一旦识别到队伍/出战宝可梦信息，在本场结束前不清除

## 5. MVP Scope

### In Scope

#### 核心能力 1：截图与识别分离，1s 截图频率

**截图线程（独立）**：
- 每秒截取 1 帧，无论识别是否完成
- 最新帧存储在服务端，始终可用
- 帧率独立于识别的处理时间

**识别线程（独立）**：
- 每秒触发一次，但只处理**最新的一帧**
- 不排队处理历史帧（跳过未处理的帧）
- 识别完成后将结果写入共享状态
- 前端轮询 `/api/recognition/current` 获取最新识别结果

#### 核心能力 2：全流程对战追踪

| 阶段 | 识别内容 | 数据行为 |
|------|---------|---------|
| 选人（team_select） | 我方 6 只：名称+道具+性别 | 写入队伍列表，**本场不清除** |
| 选人（team_select） | 对方 6 只：名称（从战斗面板补上） | 写入对方队伍列表，**本场不清除** |
| 战斗（battle） | 我方出战宝可梦：名称+HP+技能+PP | 首次检测后长期存在，有变化才更新 |
| 战斗（battle） | 对方出战宝可梦：名称+HP% | 首次检测后长期存在，有变化才更新 |
| 战斗（battle） | 对战记录：HP变化、技能使用、状态变化、更换 | 实时追加，去重 |
| 结算 | 胜利/失败画面检测 | 触发数据清除，准备下一局 |

#### 核心能力 3：5 列固定布局

```
┌──────────────────────────────────────────────────────────────────────┐
│ [视频源选择] [调试] [清除数据]                             标题栏    │
├──────────┬──────────┬──────────────┬──────────┬──────────────────────┤
│ 我方队伍  │ 我方出战  │              │ 对方出战  │ 对方队伍             │
│ (6只)    │ 宝可梦   │   对战记录    │ 宝可梦   │ (6只)               │
│          │          │              │          │                     │
│ 名字     │ 名称/HP  │ 第 1 回合     │ 名称/HP% │ 名字                │
│ 道具/性别 │ 技能+PP  │ 我方派出了X   │ 已使用招  │ (缩略图匹配)         │
│          │ 种族值   │ 对方使用了Y   │ 式列表    │                     │
│          │ 克制摘要  │ X 中毒了     │          │                     │
│          │          │ X 受到了伤害  │          │                     │
│          │          │ 第 2 回合     │          │                     │
│          │          │ ...          │          │                     │
├──────────┴──────────┴──────────────┴──────────┴──────────────────────┤
│ 底部辅助信息区（属性克制详情等）                                       │
└──────────────────────────────────────────────────────────────────────┘
```

**布局规则**：
- **选人阶段**：左边「我方队伍」和右边「对方队伍」填满 6 只信息，中间「对战记录」显示「选人阶段」
- **战斗阶段**：5 列全显——左外=我方队伍、左内=我方出战宝可梦、中=对战记录、右内=对方出战宝可梦、右外=对方队伍
- **等待/结算**：中栏显示对应状态文字
- **数据持久化**：队伍和出战宝可梦信息一旦获得，在本场结束前不清除
- **零检测不动**：当某帧识别不到数据时，不清除已有信息，保持显示

#### 核心能力 4：对战记录（Battle Log）

- **生成时机**：每 1 秒轮询，检测到任何对战事件追加一条
- **事件类型**：派上场、使用技能、HP 变化（>10% 算伤害）、状态变化（中毒/麻痹/烧伤等）、更换宝可梦、回合过渡
- **去重**：同一事件在短时间内重复检测到时不重复记录
- **展示**：中栏实时滚动显示，最新在最上方
- **格式示例**：
  ```
  ⏱ 第 1 回合
  🏃 我方 派出了 振翼发
  🏃 对方 派出了 铁包袱
  ⚔️ 对方 铁包袱 使用了 冰冻之风
  💥 振翼发 受到了伤害! HP 68%
  🌀 振翼发 冻伤了
  ⏱ 第 2 回合
  ⚔️ 我方 振翼发 使用了 月亮之力
  💥 对方 铁包袱 受到了伤害! HP 45%
  ```

#### 核心能力 5：结算检测与手动清除

- **结算检测**：识别到「获胜/失败」字样后，自动清除所有数据，准备下一局
- **手动清除按钮**：在调试按钮旁边添加一个「清除数据」按钮，随时清空 UI，重新开始
- **清除范围**：队伍列表、出战宝可梦、对战记录、HP 历史等全部对局相关数据

### 核心能力 6（已实现/基线）：选人与战斗识别

详见 §14 既有功能清单。

### Out of Scope (for MVP)
- 阵容推荐器
- 选人决策算法
- 自动决策引擎
- 云端账号系统
- 多人协作
- 全自动无人工校正工作流

## 6. User Stories

### Story 1
As a 主播,
I want to 工具每秒截取最新画面、每秒独立识别一次,
So that 我看到的对战信息接近实时，不会因为 OCR 慢而卡画面。

### Story 2
As a 主播,
I want to 选人阶段我方/对方6只信息填满5列布局后，本场不清除,
So that 我随时能回头查看双方全阵容。

### Story 3
As a 主播,
I want to 对战记录持续显示谁的宝可梦使用了什么技能、谁受到了伤害、谁出了异常状态,
So that 我不需要自己记操作顺序。

### Story 4
As a 主播,
I want to 在对局结束后（结算画面）自动清数据，准备下一局,
So that 我不需要手动刷新页面。

### Story 5
As a 主播,
I want to 按一个按钮随时清除所有数据重新开始,
So that 遇到识别错乱时可以快速重置。

### Story 6
As a 主播,
I want to 5列布局中的出战宝可梦卡片在没有检测到更新时保持原有信息,
So that UI 不会频繁闪烁或清空。

## 7. Core Architecture

### Form Factor
**Local-first Web App + 本地视频识别服务**
- 本机启动 FastAPI 后端 + Next.js 前端
- 通过浏览器打开独立窗口使用
- 浏览器窗口天然适合与 OBS 并排的第二屏场景
- 未来可封装为 Electron/Tauri 桌面客户端

### 截图/识别分离架构（本次核心改进）

```
定时器 1s ──→ CaptureThread（独立线程）
                  │
                  ├── 抓帧 ──→ FrameStore（覆盖写入最新帧）
                  │
定时器 1s ──→ RecognizeThread（独立线程）
                  │
                  ├── 从 FrameStore 读取最新帧
                  ├── 识别（OCR + 阶段 + 状态追踪）
                  ├── 更新 RecognitionState
                  └── 更新 BattleStateStore
                          │
前端轮询 1s ←─── /api/recognition/current ←─── 返回最新已识别的状态
```

### Technology Stack
- **Frontend**: Next.js / React
- **Backend API**: FastAPI
- **OCR**: PaddleOCR (ONNX 模式)
- **Video Input**: OpenCV VideoCapture + FFmpeg dshow
- **Data Storage**: 本地 JSON / 静态数据结构（已预留扩展至 SQLite）
- **Frame Processing**: OpenCV

## 8. API Specification

### `GET /api/video/sources`
列出本机可用视频源。

### `POST /api/video/source/select`
选择当前识别所使用的输入源。

### `POST /api/recognition/session/start`
启动识别会话（含截图线程+识别线程）。

### `GET /api/recognition/current`
返回最新识别结果：阶段、双方名称/HP、队伍列表、对战记录、ROI payloads。

### `POST /api/recognition/override`
人工修正当前识别结果。

### `POST /api/recognition/reset`
手动清除当前对局数据，重新开始。

### `GET /api/video/latest-frame`
轻量接口：仅返回最新截图帧（不含识别），供前端实时画面预览。

## 10. Risks & Mitigations

### Risk 1: 1s 截图 + 1s 识别导致 CPU 高负载
Mitigation：
- 截图线程与识别线程独立，识别线程跳过处理中的旧帧
- OCR 已在 ONNX 模式下运行，性能较好
- 可动态调节识别频率（当前 1s，用户可配置）

### Risk 2: 结算检测不准
Mitigation：
- 增加手动清除按钮作为兜底
- 结算检测使用 OCR 识别关键文字（胜利/获胜/失败等）

### Risk 3: 对战记录去重不完善
Mitigation：
- 按事件类型+时间窗口+hash 三重去重
- 允许用户手动查看原始序列

## 12. Key Assumptions

1. 用户主要在本机使用该工具
2. 识别语言第一优先级为中文
3. OBS 虚拟摄像头为主要视频输入方式
4. 用户接受"浏览器第二窗口"作为交互方式
5. 结算检测依赖特定文字出现（胜利/获胜/失败等）
6. 选人阶段对方 6 只通过战斗面板 OCR 逐步补全，而非即时全识别
