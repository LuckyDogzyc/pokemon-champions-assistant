# Pokemon Champions Assistant — PRD

## 1. Executive Summary

Pokemon Champions Assistant 是一个面向**主播 / 内容创作者**的宝可梦对战辅助工具。

它运行在电脑端，配合 OBS、采集卡或其他视频输入方式，从 Switch 对战画面中获取信息，并在**另一个独立窗口**中实时展示辅助内容，帮助主播快速做出更合理的对战判断。

长期目标包括但不限于：
- 阵容推荐器
- 伤害计算器
- 对战决策助手
- 宝可梦资料检索
- 战术建议系统
- 多语言视频信号识别能力

MVP 第一阶段聚焦三个最小可落地能力：
1. 采集卡实时视频输入
2. 识别双方当前场上宝可梦名称并联动资料查询
3. 属性克制查询

---

## 2. Mission

打造一个适合直播场景的宝可梦对战辅助系统，让主播在观看对战画面时，能够通过旁路工具窗口快速获得可靠、清晰、低干扰的对战信息支持。

核心使命：
- **缩短主播检索信息的时间**
- **降低直播中临场决策压力**
- **为后续自动识别和实时决策打下基础**

---

## 3. Target Users

### Primary Users
- 宝可梦对战主播
- 游戏内容创作者
- 录播内容制作者

### User Characteristics
- 通常会在电脑上同时打开 OBS、浏览器、聊天工具等多个窗口
- 需要边直播边快速查看战斗信息
- 希望辅助工具尽量少打断直播节奏
- 对信息呈现的**清晰度、速度、稳定性**要求高于花哨 UI

---

## 4. MVP Scope

## In Scope

### MVP Feature 1: 采集卡实时输入、输入源选择与基础识别管线
支持从本地电脑获取对战视频输入，MVP 第一优先级为：
- 采集卡实时输入到本机后的可访问视频源

可扩展来源包括但不限于：
- OBS 虚拟摄像头 / OBS 预览输出
- 本地摄像头/视频设备接口

MVP 必须提供**输入信号选择菜单**，至少支持：
- 列出可用视频输入源
- 选择当前识别所使用的视频源
- 展示当前选中的输入源状态

第一版目标不是完整自动对战理解，而是先建立最小可用识别管线：
- 默认从**每 3 秒抓 1 帧**开始做识别，后续根据效果再提速
- 先判断当前画面所属阶段，例如：选人、换人、战斗中、技能释放
- 根据阶段选择对应的轻量识别策略
- 通过固定锚点快速定位识别区域
- 在战斗阶段优先识别**双方当前场上宝可梦名称**
- 支持后续 OCR / 图像识别扩展
- 为当前对局资料查询提供输入上下文

### MVP Feature 2: 基于当前对局的资料查询
系统应能基于当前对局画面识别出的信息，辅助展示对应资料，例如：
- 当前所处阶段（选人、换人、战斗中、技能释放）
- 我方当前场上宝可梦名称
- 对方当前场上宝可梦名称
- 对应属性
- 基础种族值
- 常见特性
- 常见定位 / 简述

同时保留手动搜索作为兜底方式。

### MVP Feature 3: 属性克制查询
支持基于识别结果或手动输入展示：
- 单属性克制关系
- 双属性组合克制关系
- 攻击端克制
- 防守端弱点 / 抗性 / 免疫

### MVP Feature 4: 适合直播场景的第二窗口界面
要求：
- 在电脑上独立窗口打开
- 可与 OBS 并行使用
- 页面布局适合直播过程中快速扫读
- 能清楚区分“识别结果”和“资料查询结果”
- 必须包含输入信号选择菜单与当前输入状态展示

### MVP Feature 5: 多语言识别预留
MVP 第一版即要考虑多语言视频信号识别的架构兼容性，但当前阶段优先级明确为：
- **识别先以中文为第一语言**
- 识别层与资料层解耦
- 支持多语言名称映射表
- 允许后续扩展中 / 英 / 日等语言识别

### MVP Feature 6: 手动校正能力
由于第一版视频识别稳定性可能有限，MVP 应允许用户手动修正当前识别对象，以确保直播时可继续使用。

### Supporting Data Capabilities
支持按名称查询宝可梦基础资料，例如：
- 名称（中 / 英 / 日等可扩展）
- 属性
- 基础种族值
- 常见特性
- 常见定位 / 简述

并为后续建立完整对战数据库预留结构，未来数据库应逐步覆盖：
- 全宝可梦基础资料
- 种族值
- 属性
- 特性
- 技能池
- 常见对战信息与可扩展字段

名称匹配层需要支持模糊匹配与别名归一化，以便服务 OCR 纠错和手动查询。

## Out of Scope (for MVP)
- 阵容推荐器
- 伤害计算器
- 自动决策引擎
- 云端账号系统
- 多人协作
- 全量高精度战局理解（如招式、伤害数值、场地效果、完整双方状态）
- 全自动无人工校正工作流

## 5. User Stories

### Story 1
As a 主播,
I want to 从输入信号菜单中选择当前采集卡视频源,
So that 工具可以读取正确的对战画面开始识别。

### Story 2
As a 主播,
I want to 让工具读取采集卡输入的实时对战视频画面，并先判断当前处于选人、换人、战斗还是技能释放阶段,
So that 系统可以根据不同阶段使用更合适的轻量识别策略。

### Story 3
As a 主播,
I want to 基于当前识别出的宝可梦查看它的属性和基础资料,
So that 我能在直播中快速解释或判断对局。

### Story 4
As a 主播,
I want to 查看某个属性或属性组合的克制关系,
So that 我能更快判断换人、出招或战术方向。

### Story 5
As a 主播,
I want to 在 OBS 旁边打开一个独立辅助窗口,
So that 我可以边直播边查看信息而不打断节奏。

### Story 6
As a 主播,
I want to 在识别不准时手动修正结果,
So that 工具在直播过程中依然可靠可用。

### Story 7 (Future)
As a 主播,
I want to 让系统更完整地识别 Switch 视频画面中的对战信息,
So that 它能主动给出更强的实时建议。

## 6. Core Architecture & Patterns

## Recommended Form Factor

基于当前需求，**推荐第一版采用“本地桌面优先的本地 Web 应用”形态**。

### 推荐方案：Local-first Web App + 本地视频识别服务
即：
- 在本机启动一个本地 Web 服务
- 同时运行一个本地视频采集 / 识别服务
- 优先接入采集卡实时输入视频源
- 前端提供输入信号选择菜单
- 识别流程先做阶段判断（选人 / 换人 / 战斗 / 技能释放）
- 再根据阶段走轻量识别分支
- 用户通过浏览器打开一个独立页面/窗口
- 该窗口与 OBS 并排使用
- 前端展示双方当前识别结果、资料卡和克制信息

### Why this is the best first choice
1. **最适合“OBS + 第二窗口”场景**
   - 浏览器窗口天然适合做第二屏工具
   - 不需要先做复杂桌面客户端打包

2. **开发速度快**
   - 适合先验证 MVP
   - 前后端拆分清晰，后期更容易扩展 OCR / 视频识别 / AI 决策

3. **未来扩展性强**
   - 后期可封装成 Electron / Tauri 桌面应用
   - 也可继续保留为本地 Web 工具

4. **能够从第一版就接入实时识别模块**
   - 视频采集、OCR、识别服务可以作为本地服务接入
   - 不必等桌面客户端成型后再做识别

### Alternative Options Considered

#### Option A: 纯桌面应用（Electron / Tauri）
优点：
- 更像正式客户端
- 窗口控制能力更强

缺点：
- 初期开发与打包复杂度更高
- MVP 阶段性价比不如本地 Web 高

#### Option B: 纯命令行工具
优点：
- 开发快

缺点：
- 不适合主播边直播边快速查看
- 可视化太弱

### Decision
**MVP 阶段采用本地 Web 应用 + 本地识别服务。**
后续如有需要，再封装成 Electron/Tauri 桌面客户端。

### Suggested High-Level Architecture
- **Frontend**: 一个适合快速查询和展示的 Web UI，包含输入源选择菜单、识别状态区和人工修正区
- **Backend API**: 提供宝可梦资料、属性克制、输入源管理与识别状态 API
- **Recognition Service**: 负责视频源接入、低频抓帧、阶段识别、锚点定位、OCR/识别编排
- **Data Layer**: 本地静态数据文件或轻量数据库，包含多语言名称映射，并为完整宝可梦对战数据库扩展预留结构
- **Future Extension Layer**:
  - 更高精度 OCR / 图像识别模块
  - 更完整的视频帧分析模块
  - 实时决策建议模块
  - 多语言识别模块

## 8. Technology Stack

## Current Recommendation
由于当前无技术偏好，建议采用：

### Frontend
- Next.js / React

理由：
- 适合快速搭界面
- 组件化适合后续扩展卡片、面板、实时状态区
- 生态成熟

### Backend
- FastAPI

理由：
- 适合快速构建查询 API
- 对 OCR、模型推理、本地服务整合友好
- Python 在图像识别、数据处理方面扩展空间大

### Data Storage
- MVP 优先：本地 JSON / YAML / CSV 数据源
- 如查询逻辑复杂化，再升级到 SQLite
- 数据结构从第一版开始就为完整宝可梦对战数据库预留字段

### Recognition / CV Layer
- 视频帧处理：OpenCV
- 默认先按每 3 秒抓 1 帧运行，再根据效果决定是否提速
- 先做阶段识别（选人 / 换人 / 战斗 / 技能释放）
- 基于阶段与固定锚点选择轻量识别路径
- OCR：优先中文 OCR，候选为 PaddleOCR / EasyOCR / Tesseract（待选型）
- 多语言名称映射：本地字典或结构化数据表

### Future Capabilities
- AI 建议系统：可后续接 LLM 或规则引擎

## 10. API Specification

### MVP Candidate APIs

#### `GET /api/pokemon/search?q=<name>`
根据名称搜索宝可梦。

#### `GET /api/pokemon/:id`
获取某只宝可梦详细资料。

#### `GET /api/type/:typeName/matchups`
获取单属性克制关系。

#### `POST /api/type/combined-matchups`
输入双属性，返回组合克制关系。

Request example:
```json
{
  "types": ["Fire", "Flying"]
}
```

#### `GET /api/video/sources`
列出本机可用视频源。

#### `POST /api/video/source/select`
选择当前识别所使用的输入源。

#### `POST /api/recognition/session/start`
启动识别会话。

#### `GET /api/recognition/current`
返回当前识别到的我方/对方场上对象、当前阶段及置信信息。

#### `POST /api/recognition/override`
人工修正当前识别结果。

2. 用户可以从输入信号菜单中选择采集卡视频源

## 14. Risks & Mitigations

### Risk 1: 一开始目标过大
问题：长期愿景很多，容易导致项目起步过重。

Mitigation：
- 严格把第一版限制在“视频接入 + 当前对象资料查询 + 属性克制查询”

### Risk 2: 视频识别比预期更复杂
问题：OCR、多语言、实时识别复杂度高，第一版准确率可能不足。

Mitigation：
- 将 MVP 识别目标限制为“最小可用链路”
- 先做识别 + 人工修正双轨制
- 先以“每 3 秒 1 帧”控制识别节奏，稳定后再提速
- 先做阶段识别，再按阶段进入轻量识别流程
- 优先识别最关键、最稳定的信息区域

### Risk 3: 直播场景下 UI 不实用
问题：普通网页 UI 不一定适合主播高压场景。

Mitigation：
- 第一版就围绕“快速扫读”和“第二窗口使用”设计

### Risk 4: 数据源不统一
问题：宝可梦资料、属性数据版本可能混乱。

Mitigation：
- MVP 阶段先固定一套结构化本地数据源

### Risk 5: 视频源兼容性差异
问题：不同采集卡、OBS 配置、系统设备路径可能不同。

Mitigation：
- 第一版优先支持最常见的采集卡视频输入方式
- 把视频输入做成可配置层
- 保留可扩展到 OBS / 其他视频源的接口

## 15. Appendix

## Key Assumptions
1. 用户主要在本机使用该工具
2. MVP 需要做最小自动识别能力，但允许人工修正
3. MVP 第一版优先接采集卡实时输入，并识别双方当前场上宝可梦名称
4. 识别语言第一优先级为中文
5. 识别默认从每 3 秒 1 帧开始，后续再根据效果提速
6. 系统需要先判断当前阶段（选人 / 换人 / 战斗 / 技能释放），再走对应轻量识别路径
7. 用户接受“浏览器第二窗口”作为第一版交互方式
8. 未来存在升级为桌面客户端的可能
9. 后续需要建设完整宝可梦对战数据库，并支持名称模糊匹配

## Proposed Decision Summary
- **产品定位**：主播用宝可梦对战辅助工具
- **MVP 能力**：采集卡实时输入 + 输入源选择菜单 + 阶段识别 + 双方当前场上宝可梦名称识别 + 资料查询 + 属性克制查询 + 人工修正
- **首发形态**：本地 Web 应用 + 本地识别服务
- **建议技术栈**：Next.js + FastAPI + OpenCV/中文 OCR + 本地静态数据（后续扩展为完整宝可梦对战数据库）
